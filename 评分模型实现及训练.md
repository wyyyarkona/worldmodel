# 评分模型实现及训练

## 1. 文档目的

这份文档整理当前 `score_model_v2` 的实际实现状态，覆盖两部分：

1. 评分模型本身的网络架构与数据流
2. 训练脚本 `train_v2.py` 的训练组织方式、多卡方式与断点恢复方式

本文档以当前仓库代码为准，不是最初方案草稿的复述。

---

## 2. 整体目标

当前评分模型是一个 **pairwise score model**。训练时输入一对候选样本 `(f1, f2)`，再结合文本条件与图像条件，让模型输出这对样本中“谁更好”的偏好信号。

训练目标不是直接做绝对分数回归，而是学习排序关系：

- 输入：候选 A、候选 B、文本条件、图像条件、阶段信息
- 输出：一个 logit / score
- 监督：teacher 给出的 `score_a` 与 `score_b`

模型学习的是：

- A 比 B 更好的概率
- 或者说，在给定条件下，这两个候选 latent 哪个更优

---

## 3. 输入数据与 manifest 约定

训练数据来自 pairwise manifest，一条样本通常包含：

- `z_t_a_path` 或 `f1_path`
- `z_t_b_path` 或 `f2_path`
- `context_path` 或 `text_emb_path`
- `clip_fea_path` 或 `image_emb_path`
- `score_a`
- `score_b`
- `weight`
- `t_idx` / `step_idx` / `step` / `step_id`

当前数据集实现 `PairwiseLatentDatasetV2` 同时兼容两套字段命名：

- 旧命名：
  - `f1_path`
  - `f2_path`
- 当前数据管线命名：
  - `z_t_a_path`
  - `z_t_b_path`

文本与图像条件字段也兼容两套：

- 文本：
  - `text_emb_path`
  - `context_path`
- 图像：
  - `image_emb_path`
  - `clip_fea_path`

## 4. 模型总览

当前主模型类是：

- `score_model_v2/models/score_model_v2.py`
- 类名：`ScoreModelV2`

整体结构可以概括为：

1. 将两个视频 latent `f1 / f2` 编码成 token 序列
2. 将文本 embedding 与图像 embedding 投影到统一隐藏空间
3. 用 query token、stage token、segment embedding 将整条比较序列拼起来
4. 送入 Qwen comparator
5. 只取 query 部分的输出做池化
6. 通过线性头输出最终 logit 与 score

可以写成：

```text
f1 latent ----\
               -> 视频编码 -> h1 --\
f2 latent ----/                    \
                                     -> [query ; h1 ; h2 ; context ; stage]
text/image embeddings -> context ---/                 |
stage id -> stage token ------------------------------/
                                                       -> QwenComparator
                                                       -> query hidden
                                                       -> pooling
                                                       -> score head
                                                       -> logit / sigmoid(score)
```

## 5. 视频分支

### 5.1 VideoProjector

文件：

- `score_model_v2/models/projectors.py`

模块：

- `VideoProjector`
- `SpatialProjector`
- `TemporalProjector`

视频 latent 的输入形状是：

```text
[B, C, T, H, W]
```

默认配置下：

- `C = 16`
- latent 时间长度通常接近 `21`
- 空间大致是 `60 x 104`

`VideoProjector` 的处理过程是：

1. `Conv3d` 做 patch embedding
2. 空间上做 `2x2` merge
3. 时间上两帧一组合并

因此视频 token 压缩是分三步完成的：

### 第一步：patch embedding

`patch_embed = Conv3d(latent_dim -> patch_dim)`

默认：

- `latent_dim = 16`
- `patch_dim = 1536`

这一步保持时间维，先降低空间分辨率。

### 第二步：SpatialProjector

把每个 `2x2` 空间邻域合并到 channel 维，再通过两层 MLP 投影到统一隐藏维度。

默认输出隐藏维：

- `hidden_dim = 2048`

### 第三步：TemporalProjector

把相邻两帧合并成一个时间 token，再通过两层 MLP 保持到 `2048` 维。

如果时间维是奇数，会复制最后一帧进行补齐。

### 当前实现的一个重要点

不同样本的 latent 时间长度可能相差 1，例如：

- `[16, 21, 60, 104]`
- `[16, 20, 60, 104]`

这是允许的。当前 `collate_batch()` 已经支持对 batch 内视频 latent 做动态 padding，而不是假设所有样本天然同 shape。

## 6. 位置编码与辅助 embedding

文件：

- `score_model_v2/models/embeddings.py`

包含 4 类 embedding：

### 6.1 Learned3DPositionEmbedding

对压缩后的视频 3D 网格加入可学习的位置编码：

- 时间位置 `pos_t`
- 高度位置 `pos_h`
- 宽度位置 `pos_w`

输入形状：

```text
[B, T, H, W, C]
```

输出形状：

```text
[B, T*H*W, C]
```

当前实现已经改成 **按实际输入动态裁切位置表**，而不是死板要求固定：

- `T=11`
- `H=15`
- `W=26`

因此如果 batch 里实际是：

- `T=10`
- 或 `T=11`

都能兼容，只要不超过配置中位置表的上限。

### 6.2 SegmentEmbedding

用于告诉 comparator 每段 token 的语义来源。当前定义的 segment 有：

- `query`
- `h1`
- `h2`
- `context`
- `stage`

这能帮助 comparator 区分：

- 哪些 token 是候选 A
- 哪些 token 是候选 B
- 哪些 token 是条件
- 哪些 token 是查询探针

### 6.3 StageEmbedding

把 `stage_id` 映射成单个 stage token。

当前一共有 3 个 stage：

- `0`
- `1`
- `2`

它们并不是训练大阶段 `warmup/lora/curriculum`，而是数据样本所属的去噪阶段 bucket。

### 6.4 QueryEmbedding

一组可学习 query token，当前默认数量：

- `num_query_tokens = 4`

最终只使用 query token 对应的输出 hidden states 来做打分。

## 7. 条件分支

文件：

- `score_model_v2/models/projectors.py`

模块：

- `ContextProjector`

输入有两部分：

1. 文本 embedding
2. 图像 embedding

### 7.1 文本分支

默认：

- `text_dim = 4096`

文本 token 数超过 `512` 时会被截断到前 `512` 个，再投影到 `2048` 维。

### 7.2 图像分支

默认：

- `image_dim = 1280`

图像 token 数超过 `64` 时，会先通过 `adaptive_avg_pool1d` 压到 `64` 个，再投影到 `2048` 维。

### 7.3 context 拼接顺序

当前实现里，context 顺序是：

```text
[image_tokens ; text_tokens]
```

---

## 8. 序列构造方式

文件：

- `score_model_v2/models/score_model_v2.py`

`build_sequence()` 会将所有 token 拼成一条完整序列，顺序严格为：

```text
[query ; h1 ; h2 ; context ; stage]
```

其中：

- `query`：可学习 query token
- `h1`：候选 A 的视频 token
- `h2`：候选 B 的视频 token
- `context`：图像 + 文本条件 token
- `stage`：当前样本所在去噪 bucket 的 stage token

每一段在拼接前都会加上对应的 `SegmentEmbedding`。

同时生成全 1 的 `attention_mask`，表示整条序列都有效。

## 9. QwenComparator

文件：

- `score_model_v2/models/qwen_comparator.py`

这是当前模型里最核心的比较器模块。

### 9.1 Comparator 的设计目标

它不是用整个 Qwen2.5-VL 做生成，也不是输入 token id，而是：

- 直接输入已经投影好的隐藏状态序列
- 只取 Qwen 语言模型前若干层
- 让这些层承担“比较器”的角色

### 9.2 当前实现方式

当前实现会：

1. 加载 `Qwen2.5-VL` 主模型
2. 找到内部 language model
3. 取前 `num_qwen_layers` 层 decoder block
4. 默认冻结 backbone
5. 在需要时挂 LoRA

默认配置：

- `num_qwen_layers = 6`

### 9.3 双向注意力

原始 Qwen 是 decoder-style causal attention。  
为了做比较任务，当前实现会 patch 掉 causal mask 更新逻辑，使得比较序列可以做 **双向注意力**。

这意味着：

- query、候选 A、候选 B、context、stage
- 都可以相互看见

### 9.4 Rotary 位置编码兼容

Qwen2.5-VL 的 block 在运行时依赖 rotary position embeddings。  
当前 comparator 实现已经把这部分补齐：

1. 从 Qwen 主体里抓取 `rotary_emb`
2. 在 comparator 前向中构造 `position_ids`
3. 构造 `position_embeddings`
4. 传给每一层 decoder block

当前 comparator 没有真实图文空间布局，因此使用同一组单调递增索引扩展成 3 轴：

```text
[3, batch, seq]
```

这是为兼容 Qwen2.5-VL 的 multimodal rotary 预期输入格式。

### 9.5 dtype / device 对齐

当前实现已经处理了多处 dtype / device 对齐问题：

- 输入到 comparator 前，对齐到 comparator 参数的 dtype/device
- 最后进入 `score_head` 前，对齐到 head 参数的 dtype/device

这样可以避免：

- `Float` vs `BFloat16`
- CPU init / GPU run

等常见错误。

### 9.6 Flash Attention 2

默认配置：

```yaml
attn_implementation: flash_attention_2
```

当前实现里，如果选择 `flash_attention_2` 且机器有 CUDA，会在 `from_pretrained()` 时直接通过 `device_map` 把模型加载到当前 GPU，以降低初始化 warning。

## 10. LoRA 实现

LoRA 也是在 `QwenComparator` 内部完成的。

### 10.1 当前挂载方式

当前实现不是把 LoRA 只挂到局部层列表，而是：

- 优先挂到 `self.full_model`
- 然后重新刷新 backbone 引用

这比只挂局部 `ModuleList(self.layers)` 更稳定，也更符合 PEFT 的使用方式。

### 10.2 当前 target_modules

LoRA 目标模块包括：

- `q_proj`
- `k_proj`
- `v_proj`
- `o_proj`
- `gate_proj`
- `up_proj`
- `down_proj`

### 10.3 当前行为是严格模式

如果出现以下任一情况：

- `peft` 没安装
- LoRA attach 失败
- attach 后没有任何 `lora_` 参数

训练会 **直接报错**，而不是再悄悄退化成全参训练。

这是当前实现里一个很重要的行为变化。

## 11. 输出头

Qwen comparator 输出整条序列的 hidden states 后，只取 query 段：

```python
query_hidden = hidden_states[:, :num_query_tokens]
```

然后：

1. 对 query token 做平均池化
2. 经过 `LayerNorm`
3. 经过 `Linear(hidden_dim -> 1)`
4. 得到 `logit`
5. 再经过 `sigmoid` 得到 `score`

因此最终监督主要作用在：

- query token 的比较结果聚合上

## 12. 损失函数

文件：

- `score_model_v2/losses.py`

### 12.1 PairwiseScoreLoss

核心损失是一个 **weighted soft-label BCE**。

做法如下：

1. 先计算 teacher 分数差：

```text
delta = score_a - score_b
```

2. 过滤掉 margin 太小的样本：

```text
|delta| < margin
```

这些 pair 被视为排序信号不稳定，会被忽略。

3. 把 `delta / tau` 过 `sigmoid`，得到 soft target

4. 用 `binary_cross_entropy_with_logits` 训练模型输出 logit

5. 权重来自：

- `|delta|` 自动生成的强弱权重
- 可选 `sample_weight`

默认配置：

- `tau = 0.3`
- `margin = 0.05`

### 12.2 WarmupAlignmentLoss

只在 `warmup` 阶段启用。

它使用 projector 输出 token 的统计量：

- `proj_mean`
- `proj_std`

与目标统计量做 MSE，对 projector 和输入对齐模块起到辅助约束作用。

默认权重：

- `warmup_alignment_weight = 0.1`

## 13. 训练数据集与 batch 组织

文件：

- `score_model_v2/train_v2.py`

### 13.1 数据集类

数据集实现：

- `PairwiseLatentDatasetV2`

主要功能：

1. 读取 `jsonl/json` manifest
2. 解析当前样本所属 stage bucket
3. 根据 curriculum 过滤样本
4. 读取 latent / text / image tensor

### 13.2 stage bucket 的来源

优先级如下：

1. 如果 manifest 里已经有显式 `stage_id`
2. 否则根据：
   - `t_idx`
   - `step_idx`
   - `step`
   - `step_id`

计算 step index 的全局范围，再切成 3 段：

- 前 1/3 -> stage 0
- 中间 1/3 -> stage 1
- 后 1/3 -> stage 2

也就是说，训练里的 `stage_id` 是按去噪 step 区间构造出来的。

### 13.3 collate 逻辑

当前 batch collate 分两类处理：

#### token 类

- `text_emb`
- `image_emb`

使用 `pad_token_batch()` 做右侧 padding。

#### 视频 latent 类

- `f1`
- `f2`

使用 `pad_video_latent_batch()` 按 batch 内最大：

- `T`
- `H`
- `W`

做零填充。

这是当前实现中一个关键兼容点，因为不同 pair 虽然内部 shape 一致，但不同 pair 之间可能出现：

- `T=20`
- `T=21`

## 14. 训练阶段设计

训练入口：

- `score_model_v2/train_v2.py`

当前训练是 **串行阶段训练**，不是 warmup 和 lora 交替执行。

阶段顺序固定为：

1. `warmup`
2. `lora`
3. `curriculum`

每个阶段会先跑完自己的所有 epoch，再进入下一个阶段。

也就是：

```text
warmup(全部轮次) -> lora(全部轮次) -> curriculum(全部轮次)
```

不是：

```text
warmup1 -> lora1 -> warmup2 -> lora2
```

### 14.1 warmup 阶段

训练参数只开放这些新模块：

- `video_projector`
- `video_pos_embed`
- `context_projector`
- `segment_embed`
- `stage_embed`
- `query_embed`
- `score_norm`
- `score_head`

Qwen backbone 保持冻结。

### 14.2 lora 阶段

在 warmup 的基础上，再开启 comparator 中的 LoRA 参数训练。

也就是：

- 新模块继续训练
- `lora_` 参数也训练

### 14.3 curriculum 阶段

`curriculum` 阶段会根据配置里的：

```yaml
curriculum_stage_ids: [2, 1, 0]
```

拆成多个 phase，例如：

- phase 1 -> 只训练 stage 2 样本
- phase 2 -> 只训练 stage 1 样本
- phase 3 -> 只训练 stage 0 样本

如果命令行加：

```bash
--disable_curriculum
```

则训练会把：

```yaml
epochs.curriculum = 0
```

也就是直接跳过第三阶段。

## 15. 优化器策略

`build_optimizer()` 当前策略如下：

### warmup

所有可训练参数统一使用：

- `warmup_lr`

### lora / curriculum

参数会分成两组：

1. `lora_` 参数
   - 使用 `lora_lr`
2. 其他仍可训练的新模块
   - 使用 `base_lr`

默认配置：

- `warmup_lr = 1e-4`
- `lora_lr = 5e-5`
- `base_lr = 1e-4`
- `weight_decay = 1e-2`

## 16. 单卡与多卡训练

### 16.1 分布式入口

当前训练脚本同时支持：

- 单进程单卡
- `torchrun` 多卡 DDP

分布式初始化由 `init_distributed()` 完成。

### 16.2 DDP 包装

训练进入每个阶段后，如果是多卡，会用：

- `DistributedDataParallel`

包装当前模型。

当前支持命令行参数：

```bash
--ddp_find_unused_parameters
```

这个参数在当前 staged training 里很重要，因为不同阶段有些参数会被冻结，或者某轮不参与 loss。

### 16.3 Sampler

多卡训练使用：

- `DistributedSampler(train_dataset, shuffle=True)`
- `DistributedSampler(val_dataset, shuffle=False)`

每轮训练前会：

```python
train_sampler.set_epoch(...)
```

保证每个 epoch 的 shuffle 一致但不同。

## 18. checkpoint 与断点恢复

### 18.1 恢复入口

当前恢复逻辑优先级是：

1. `--resume_from`
2. 如果没传且没加 `--no_auto_resume`
3. 自动尝试读取：

```text
output_dir/latest_checkpoint.pt
```

### 18.2 当前保存策略

当前实现已经改成：

- `latest_checkpoint.pt`：每个 epoch 都更新
- 独立 checkpoint：按 `checkpoint_interval_epochs` 保存

默认配置：

```yaml
checkpoint_interval_epochs: 5
```

因此：

- 断点恢复主要依赖 `latest_checkpoint.pt`
- 历史回滚依赖周期性保存的独立 checkpoint

### 18.3 checkpoint 命名

独立 checkpoint 目录命名带 `phase`，例如：

- `warmup_phase1_epoch_005`
- `lora_phase1_epoch_010`
- `curriculum_phase2_epoch_005`

### 18.4 当前恢复逻辑的兼容性

当前实现已经兼容旧 checkpoint 中保存的：

- `stage = lora_phase1`

这类格式，会先归一化成：

- `lora`

再参与阶段比较与 epoch 跳过判断。

## 21. 现阶段实现特点总结

当前 `score_model_v2` 的特点可以总结为：

1. **模型上**
   - 视频 latent、文本条件、图像条件统一投影到 `2048` 维
   - 使用 query token + Qwen 前几层做比较器
   - 最终只用 query hidden 做打分

2. **训练上**
   - 采用 `warmup -> lora -> curriculum` 的串行阶段训练
   - 支持 `--disable_curriculum`
   - 支持单卡和多卡 DDP

3. **工程上**
   - 支持 repo-relative 路径
   - 支持旧/新 manifest 字段名
   - 支持 batch 内 variable-length latent padding
   - 支持带 phase 的 checkpoint 命名与自动恢复
   - 支持只保留 `latest` 高频恢复、独立 checkpoint 低频保存

4. **约束上**
   - LoRA attach 失败时直接报错
   - 不再退化成静默全参训练
