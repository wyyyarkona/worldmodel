# Score Model V2 诊断手册

本文档总结当前 `score_model_v2` 在训练与验证过程中，用于定位“模型没有学会比较”“输出塌成常数”“query readout 失效”等问题的完整诊断流程。

内容包括：

- 诊断思路
- 使用到的脚本
- 服务器执行命令
- 结果怎么看
- 常见现象对应的结论

## 1. 什么时候需要进入诊断流程

如果你在训练或验证时出现下面这些现象，就应该进入诊断流程：

- `val_pairwise_accuracy` 长期在 `0.5`
- `val_weighted_pairwise_accuracy` 长期在 `0.5`
- `val_auc` 长期在 `0.5`
- `val_accuracy_by_step` 所有 step 都是 `0.5`
- `pred_prob` 在整批样本上几乎是常数
- 交换 `(f1, f2)` 后输出几乎不变

这些现象通常说明：

- 模型没有学到有效比较能力
- 或者输出退化成了常数分类器

## 2. 诊断总思路

当前推荐按下面顺序排查。

### 第一步：先看 epoch 级指标

先确认问题是不是“真的没有学到”。

重点看：

- `val_pairwise_accuracy`
- `val_weighted_pairwise_accuracy`
- `val_auc`
- `val_accuracy_by_step`
- `val_accuracy_by_margin_bucket`

如果这些都接近随机水平，再进入下一步。

### 第二步：看样本级预测是不是塌成常数

重点看：

- `pred_prob`
- `pred_logit`
- `pred_label_counts`

如果发现：

- 所有样本的 `pred_prob` 都一样
- 所有样本的 `pred_label` 都一样

那么说明模型输出已经塌成常数。

### 第三步：确认是不是数据没信号

这一步不能靠正式模型判断，而要靠 baseline 或样本统计判断。

重点看：

- `teacher_delta`
- `margin`
- baseline 是否能学起来

如果 baseline 很快就能达到较高精度，说明：

- 数据本身有信号
- 问题更可能在正式模型结构

### 第四步：检查结构内部哪一层塌了

重点排查：

1. `score_head.weight` / `score_head.bias` 是否只是学了偏置
2. `h1` / `h2` 是否本身有差异
3. `query_hidden` / `pooled` 是否跨样本完全不变
4. 交换 `(f1, f2)` 后输出是否变化

如果：

- `h1/h2` 在变
- `h1_hidden/h2_hidden` 也在变
- 但 `query_hidden/pooled/logit` 不变

那么当前最可能的问题就是：

- comparator 在处理 token
- 但 query readout 没有读到比较信息

## 3. 诊断用到的脚本

当前诊断流程主要用下面这些脚本。

### 3.1 `plot_training_metrics.py`

用途：

- 画训练曲线和验证指标曲线

主要看：

- `train_loss / val_loss`
- `val_pairwise_accuracy`
- `val_weighted_pairwise_accuracy`
- `val_auc`
- `val_brier_score`
- `val_accuracy_by_stage`
- `val_accuracy_by_step`
- `margin bucket` 曲线

### 3.2 `analyze_predictions.py`

用途：

- 对某一轮验证导出的 `*_val_predictions.jsonl` 做统计

主要输出：

- `pred_prob` 分布
- `pred_logit` 分布
- `pred_label_counts`
- `target_label_counts`
- `teacher_delta`
- `margin`
- `accuracy_by_step`
- `accuracy_by_stage`
- `accuracy_by_margin_bucket`

### 3.3 `eval_manifest.py`

用途：

- 用一个已经训练好的 checkpoint，对任意 manifest 跑纯评估

适合用来回答：

- 训练集上是不是也塌成常数输出
- 某个新构建的验证集上表现如何

### 3.4 `debug_model_diagnostics.py`

用途：

- 做结构级调试

会检查：

- `score_head.weight / bias`
- `h1`
- `h2`
- `h1_vs_h2`
- `context`
- `query_input`
- `query_hidden`
- `h1_hidden`
- `h2_hidden`
- `context_hidden`
- `pooled_before_norm`
- `pooled_after_norm`
- 交换 `(f1, f2)` 前后各部分输出差异

这是定位 comparator / query readout 问题的关键脚本。

### 3.5 `baseline_probe.py`

用途：

- 做最小 sanity-check baseline

目标：

- 验证数据本身是不是可分

### 3.6 `baseline_train_ddp.py`

用途：

- 做正式 baseline 对照实验

目标：

- 验证简单 pairwise 结构能不能学起来
- 给正式模型提供强对照

## 4. 训练完成后的第一步：看训练曲线

### 命令

```bash
cd /data/zhulanyun/lihailong/repo

python -m score_model_v2.plot_training_metrics \
  --run_dir /data/zhulanyun/lihailong/outputs/你的输出目录
```

### 重点怎么看

#### 如果看到：

- `val_pairwise_accuracy ≈ 0.5`
- `val_auc ≈ 0.5`
- `val_accuracy_by_step` 全部接近 `0.5`

结论：

- 模型没有学会比较

#### 如果看到：

- `train_loss` 降了
- 但 `val_pairwise_accuracy / val_auc` 不动

结论：

- 模型在拟合某种数值目标
- 但没有真正形成排序能力

## 5. 第二步：检查样本级预测是不是常数输出

### 命令

先找某一轮验证预测文件：

```bash
find /data/zhulanyun/lihailong/outputs/你的输出目录/predictions -name "*_val_predictions.jsonl"
```

然后统计：

```bash
cd /data/zhulanyun/lihailong/repo
source /data/zhulanyun/lihailong/venv_unified310/bin/activate

python -m score_model_v2.analyze_predictions \
  --predictions /data/zhulanyun/lihailong/outputs/你的输出目录/predictions/某个_val_predictions.jsonl
```

### 重点怎么看

#### 关键字段 1：`pred_label_counts`

如果出现：

```json
"pred_label_counts": {
  "0": 504
}
```

或者：

```json
"pred_label_counts": {
  "1": 504
}
```

说明：

- 所有样本都被判成同一类

#### 关键字段 2：`pred_prob`

如果：

- `min = p25 = mean = median = p75 = max`

说明：

- 所有样本的预测概率完全一样
- 输出已经塌成常数

#### 关键字段 3：`teacher_delta` 和 `margin`

如果它们分布正常，而 `pred_prob` 仍是常数，说明：

- 问题不在 teacher 标签
- 问题更可能在模型结构或读出方式

## 6. 第三步：看训练集上是不是也塌了

如果你怀疑：

- 训练集和验证集一样都塌成常数

可以用 `eval_manifest.py` 直接对训练集做评估。

### 命令

```bash
cd /data/zhulanyun/lihailong/repo
source /data/zhulanyun/lihailong/venv_unified310/bin/activate

python -m score_model_v2.eval_manifest \
  --config score_model_v2/configs/v2_default.yaml \
  --checkpoint /data/zhulanyun/lihailong/outputs/你的输出目录/latest_checkpoint.pt \
  --manifest /data/zhulanyun/lihailong/data/score_model_data/openvid_i2v_81f/manifests/pairs_train.dev4500.jsonl \
  --output_dir /data/zhulanyun/lihailong/outputs/你的输出目录/train_eval \
  --name train_eval
```

然后再统计：

```bash
python -m score_model_v2.analyze_predictions \
  --predictions /data/zhulanyun/lihailong/outputs/你的输出目录/train_eval/predictions/train_eval_phase1_epoch_001_val_predictions.jsonl
```

### 重点怎么看

如果训练集上也出现：

- `pred_prob` 常数
- `pred_label_counts` 全部为同一类

说明：

- 不是泛化失败
- 是模型本身没有学到区分能力

## 7. 第四步：做结构级诊断

这是定位问题最关键的一步。

### 命令

```bash
cd /data/zhulanyun/lihailong/repo
source /data/zhulanyun/lihailong/venv_unified310/bin/activate

python -m score_model_v2.debug_model_diagnostics \
  --config score_model_v2/configs/v2_default.yaml \
  --checkpoint /data/zhulanyun/lihailong/outputs/你的输出目录/latest_checkpoint.pt \
  --manifest /data/zhulanyun/lihailong/data/score_model_data/openvid_i2v_81f/manifests/pairs_val.dev500.jsonl \
  --batch_size 8 \
  --max_samples 32 \
  --output /data/zhulanyun/lihailong/outputs/你的输出目录/diagnostics/val_debug.json
```

### 结果如何看

#### 7.1 `score_head_weight` / `score_head_bias`

看什么：

- `weight` 是否全零
- `bias` 是否只是学成了一个固定偏置

如果：

- `weight` 正常
- 但输入给 head 的特征是常数

那 head 也只能输出常数。

#### 7.2 `h1` / `h2`

看什么：

- `variation_across_samples.mean_per_feature_std_across_batch`

如果这里明显大于 0，说明：

- 不同样本的视频分支输出是有差异的

#### 7.3 `h1_vs_h2`

看什么：

- `global_mean_abs_diff`

如果明显大于 0，说明：

- 同一样本内的两个候选视频本身是不同的

这可以排除：

- 输入候选完全一样

#### 7.4 `query_hidden`

看什么：

- `variation_across_samples.mean_per_feature_std_across_batch`

如果这里等于 0，说明：

- comparator 输出给 query 的结果对所有样本都一样

这是严重异常。

#### 7.5 `pooled_before_norm` / `pooled_after_norm`

如果这两个的跨样本变化都等于 0，说明：

- 送进最终打分头的是常数特征

那么：

- `logit` 也只能是常数

#### 7.6 `swap_test`

重点看：

- `logit_abs_diff_mean`
- `query_hidden_abs_diff`
- `pooled_abs_diff`

如果交换 `(f1, f2)` 后这些都等于 0，说明：

- 模型对顺序完全不敏感
- 没有真正学会比较

### 已经观察到的一种典型现象

如果你看到：

- `h1` / `h2` 有变化
- `h1_hidden` / `h2_hidden` 也有变化
- 但 `query_hidden` / `pooled` / `logit` 完全不变

那么结论通常是：

- comparator 在处理 token
- 但 query readout 没有读到比较信息

## 8. 第五步：用 baseline 判断数据本身是否可分

如果你不确定是：

- 数据没信号
- 还是正式模型结构有问题

可以用 baseline。

### 8.1 最小 probe

脚本：

- `baseline_probe.py`

它的结构非常简单：

```text
f1 -> VideoProjector -> mean -> h1_pool
f2 -> VideoProjector -> mean -> h2_pool
(h1_pool - h2_pool) -> LayerNorm -> Linear -> Sigmoid
```

如果这个都能学起来，说明：

- 数据和 latent 特征本身是有信号的

### 8.2 正式 baseline

脚本：

- `baseline_train_ddp.py`

它支持两种结构。

#### 结构 A

```text
f1 -> VideoProjector -> mean -> h1_pool
f2 -> VideoProjector -> mean -> h2_pool
[h1_pool ; h2_pool ; h1_pool - h2_pool ; h1_pool * h2_pool] -> MLP -> score
```

#### 结构 B

```text
f1 -> VideoProjector -> mean -> h1_pool
f2 -> VideoProjector -> mean -> h2_pool
(h1_pool - h2_pool) -> LayerNorm -> Linear -> score
```

### baseline 能说明什么

如果 baseline 很快就达到较高 `val_accuracy`，说明：

- 数据本身可分
- 视频特征本身有信号
- 正式模型失败更可能是 comparator / readout 的问题

## 10. 常见现象与结论对照

### 现象 1

- `pred_prob` 是常数
- `pred_label_counts` 只有一类

结论：

- 输出塌成常数分类器

### 现象 2

- `teacher_delta` / `margin` 分布正常
- 但 `pred_prob` 仍然是常数

结论：

- 问题不在标签，问题在模型

### 现象 3

- `h1_vs_h2 > 0`
- `query_hidden` 跨样本变化 = 0

结论：

- 输入候选有差异
- 但 query 没读到这些差异

### 现象 4

- `h1_hidden` / `h2_hidden` 跨样本变化明显
- `pooled` 跨样本变化 = 0

结论：

- comparator 在处理 video token
- 但最终 readout 失效

### 现象 5

- 交换 `(f1, f2)` 后 `logit_abs_diff_mean = 0`

结论：

- 模型没有真正学习到方向性比较

### 现象 6

- baseline 很快达到较高准确率
- 正式模型仍然塌成常数

结论：

- 问题主要不在数据
- 更可能在正式模型结构，尤其是 comparator / query readout

## 11. 当前最推荐的诊断顺序

建议按这个顺序做：

1. 看 `metrics.json` 和可视化图
2. 用 `analyze_predictions.py` 检查是否常数输出
3. 用 `eval_manifest.py` 检查训练集是否也塌
4. 用 `debug_model_diagnostics.py` 定位结构内部问题
5. 用 baseline 验证数据是否可分
6. 如果确认是 query readout 失效，再考虑改模型读出方式

## 12. 一句话总结

当前这套诊断流程的核心思想是：

- 先确认是不是“真的没学到”
- 再确认是不是“塌成常数输出”
- 再排除“数据本身没信号”
- 最后用结构诊断定位是 comparator、query readout，还是更下游的 head 出了问题
